The World Wide Web turned 30 years old this year back in March. Maybe this is considered ancient for some of you, but if you think about it, this technology that almost all of us use everyday, is really quite young.

Web layouts over the years

So when the web started out, there was no layout. You could format your text with spaces, line breaks and indents, sure. But having a designed visual layout on your web page? Nope.

The first time we could sort of place things in specific positions on a webpage was probably üëæ in 1994 when table elements were supported in browsers. Originally meant for people to make data tables, they were soon misused for layouts. Because developers discovered‚Ä¶ üëæ nesting.

Then üëæ CSS came along in 1996, and with it came floats. And we learned a new way to layout elements on their page. üëæ But floats weren't easy to deal with. Columns were uneven heights, things got displaced and didn't line up, you had to clear stuff. üëæ

And because it was so hard, üëæ some people decided to build frameworks, and let everyone else apply these pre-written styles with CSS classes. This approach was really popular and we hit an age where a lot of websites had very similar layouts.

üëæ But during that time, CSS also started to have properties that were designed to suit the web better, to suit the fact that content on the web is very different from any other static medium.

Think of CSS as an evolving technology. It started out in 1996 as a long monolithic document, and at the time, browser vendors treated the implementation of CSS a bit like how we Malaysians treat traffic laws. We take them as suggestions rather than rules.

The CSS Working Group was created in 1997 to tackle issues which were not addressed in the first version of CSS and they have been working on improving and evolving CSS ever since.

By 2000, CSS had been broken up into modules to clarify the relationships between different parts of the specification, and make it easier to test, update and maintain the specification as a whole.

The 2018 CSS Snapshot released in January this year now consists of 99 modules, many of which are being actively worked on.

Individual CSS properties can be quite useful on their own, but the full power of CSS truly shines through when we use different properties in combination.

For example, this highlighted subset consists of modules that are layout-related, but there are so many others as well. Some are completely new, some are enhancements on older features.

Today, I'm going to focus mostly on CSS for layouts. Not only because I genuinely enjoy moving boxes around on the page, but also, I feel it's one of those aspects of CSS where the evolution is most ah-maz-ing.

In CSS2, we had 4 layout modes for different use cases. We had block layout, which was meant for laying out boxes on the page. üëæ We had inline layout, meant for laying out text.

üëæ Then we had table layouts that worked best for laying out 2D data in a tabular format. üëæ And positioned layout, which was like an escape hatch, that let us position elements without regard to other elements on the page.

Flexible box layout

describes a CSS box model optimised for user interface design and designed for laying out more complex applications and webpages

In 2009, the flex layout model was proposed, and it was the first layout model designed specially for the web. Because on the web, we don't control how our users view our sites. So what we needed, was flexibility. And Flexbox can be very flexible. It also lets us control spacing and alignment of items like never before.

Grid versus Flexbox?

⚠️ Wrong question ⚠️

Some people have asked me, so why do we need Grid then, when we already have Flexbox? If Flexbox is really as great as I say it is. I also get the question: is Grid supposed to replace Flexbox?

üëæ Wrong questions, my friends. Wrong questions. Why? Because these 2 layout techniques are not competing against each other. They are meant to be complementary.

Grid AND Flexbox

Grid and Flexbox are best friends. Like kaya and butter. Like nuts and bolts. Like Jay-Z and Beyonc√©. Either are good on their own, but together, they can transcend awesomeness.

Flexbox ‚û° single dimension

Flexbox is suited for laying out items in a single dimension, where there is no relationship between the rows and columns. But you can customise how much an item grows or shrinks in that single dimension. It's very flexible, especially once you figure out how the sizing works.

Grid ‚û° two dimensions

Grid items, on the other hand, don't need to be sized individually. When you use grid, the track sizes are determined on the parent container, while the items are merely placed into their respective areas on the grid. The grid items also know about other items in the grid.

But more than just Flexbox and Grid, there are so many different properties, both old and new for creating layouts. It's not about using one technique for everything, instead, it's about picking the best one for the job at hand.

Where does CSS come from?

  • Browser vendors
  • Companies who need certain features

But let's take another step back and talk about where CSS actually comes from. Sometimes they are proposed by browsers themselves, not much surprise there. So Google, Mozilla, Samsung, Microsoft are all fairly involved here.

There are also non-browser companies who will propose features, like Adobe, which proposed one of my favourite specifications, CSS shapes.

Often, they are proposed by members of the CSS working group.

The CSS Working Group

I would like to take the opportunity to clear up some misconceptions about the CSS working group. Sometimes I feel that people think of standards bodies like the CSS working group like this üëæ.

No, they are human beings, friendly human beings, just like you and I. I'm not that familiar with other working groups, but I've been lucky enough to sit in on a CSS working group teleconference call, and they run one of the most organised calls I've ever experienced.

Specification document statuses

Working Draft (WD)

to create a snapshot of the specification's current state and to solicit input from the W3C and the public

Last Call Working Draft (LC or LCWD)

to announce that the specification will move toward Candidate Recommendation unless significant issues are brought up

Candidate Recommendation (CR)

to express that all known issues have been resolved, and is a call for implementations

Proposed Recommendation (PR)

needs a comprehensive test suite and implementation reports proving that every feature is interoperably implemented in at least two shipping implementations

Recommendation (REC)

finally, a standard

It's during those calls when CSS specifications get discussed. Specifications go through several stages before becoming a standard. They start off as an idea, or collection of ideas, which get fleshed out and worked through.

People will raise questions about certain use-cases, or highlight issues with syntax, with logic, with implementation details, and so on. It's not a quick nor easy process. Sometimes the process takes years and years. Usually it takes someone with a vested interest to drive the specification forward.

For Grid, I think it's safe to say one of the key people who did that was Rachel Andrew.

Fun facts about CSS grid

  • Numerous proposals submitted that were never implemented
  • First proposal that stuck was one from Microsoft (for Metro UI)
  • First implementation of Grid ever was in IE10 (with vendor prefix)
  • Bloomberg hired Igalia to implement CSS grid for Blink and WebKit
  • Rachel Andrew proposed gap to solve the gutter problem

And here are a few fun facts from the CSS grid story. As the quote earlier mentioned, grid-based layouts were a goal from the start, and many proposals were submitted, just never implemented. The first one which led to an initial implementation was from Microsoft, hence the first implementation came in IE10.

Bloomberg was actually sponsored Igalia, a Spanish open source consultancy, to implement CSS grid for Blink and Webkit. So we have them to thank for Grid in Chrome and Safari. Also, there would be no gap property if it weren't for Rachel Andrew.

So in 2017, Grid had a very successful roll-out. By the end of March, all the major browsers had shipped a stable implementation which worked the same way. And by October, even Samsung Internet and Edge (RIP) also had grid support.

Grid terminology

I'll also be referring to some grid-specific terms later, so we'll cover them here. Grid lines are the horizontal and vertical lines that form the basis of the grid structure. We use a numerical index to reference the lines, and the index starts from 1. Like Lua, not C.

Each space between grid lines are called grid tracks, these are the rows and columns of your grid, and represent the spaces your items appear in.

The grid cell is a single unit of the grid and is made up of 2 adjacent row lines and 2 adjacent column lines. By default, each grid item is 1 grid cell. Many grid cells become grid area.

The best part about Grid are grid gaps, which let us add space between grid tracks. No need for awkward negative margins and padding tricks. This feature has been expanded to flex layouts as well, so if you're using Firefox 63 and beyond, you can gap your flex items too.

What's in Grid Level 2?

    Level 2 expands Grid by:
  • adding ‚Äúsubgrid‚Äù capabilities for nested grids to participate in the sizing of their parent grids
  • aspect-ratio‚Äìcontrolled gutters

Some of us might not be too familiar the the levels in CSS specifications. Long story short, the CSS spec used to be a big monolithic document for the first 2 versions, but was split up into modules in 2000 to make it easier to test, develop and maintain the specification.

This split up version was commonly referred to as CSS3. But there is and will never be CSS4 or 5 and so on, because each CSS feature is a separate document now. Usually when a feature gets too complicated, or something in there requires more discussion, it gets deferred to the next level.

Subgrid was an important feature from the start, but it needed more time to ensure robustness, hence it was deferred to Level 2. So level 2 of the grid specification covers üëæ subgrid, as well as üëæ defines the behaviour for aspect-ratio-controlled gutters. But we'll only focus on subgrid right now.

Why do we need subgrid?

There are numerous use-cases where we want our nested grids to align up to the outermost grid, for example, if you had a grid of card-style layouts with varied internal content heights. And you wanted the headers and footers for cards in the same row to all line-up.

Or if you wanted to align inline form inputs and their labels.

There was no way to do that without subgrid because the grand-child elements don't know about the outermost grid.

The subgrid syntax

Same properties, more values

grid-template-columns: subgrid <line-name-list>?
grid-template-rows: subgrid <line-name-list>?

The subgrid keyword indicates to the browser the nested grid will use the same sizing as its parent along the relevant axis.

.subgrid-container {
  grid-columns: 2 / 5; /* placement for the subgrid container itself */

  display: grid; /* you must still apply a display: grid to the subgrid */
  grid-template-columns: subgrid;
  grid-template-rows: subgrid;
}

The subgrid keyword is what allows the nested grid to participate in the sizing of outer grids. And it is possible to keep nesting subgrids as well. üëæ But maybe don't go too overboard with it. üëæ

It's important to remember that the subgrid value is used in grid-template-columns and grid-template-rows, not the display property.
